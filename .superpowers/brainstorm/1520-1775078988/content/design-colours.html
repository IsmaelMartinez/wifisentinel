Design Section 1: Accessible Colour System + Scan Progress

The foundation — these decisions affect every other section

Accessible-by-Default Colour Palette

The current reporter uses red/green as the primary pass/fail signal, which is indistinguishable for ~8% of men with colour vision deficiency. The new palette uses shape + colour + label together so no single channel carries meaning alone.

Proposed Semantic Palette
✔ PASS Teal/cyan — distinguishable from red in all CVD types Was: green ✘ FAIL Red stays — icon + label disambiguate from teal Unchanged ▲ WARN Amber/gold — triangle icon, distinct from both teal and red Was: yellow (similar) ● INFO Blue — circle icon for neutral information Was: white/default — N/A Dim grey — dash, clearly inactive Unchanged

Key principle: every status is communicated through three redundant channels — colour, icon shape, and text label. Someone seeing only greyscale can still tell pass from fail via the checkmark vs cross. The figures library provides these symbols with automatic ASCII fallbacks for limited terminals.

Scan Progress with listr2

Instead of a blank terminal during scanning, listr2 renders a live task tree that maps to the scan pipeline stages.

Terminal During Scan
✔ WiFi environment 802.11ax · 5GHz · ch48 · WPA3
✔ DNS audit 2 servers · DNSSEC enabled
✔ Security posture firewall on · VPN active
✔ Active connections 14 established · 6 listening
◤ Host discovery found 7 hosts...
○ Port scanning
○ Deep analysis (hidden devices, intrusion)
○ Speed test
Network Map
┌─ 192.168.1.1 Cisco Systems (gateway)
├─ 192.168.1.100 Apple Inc
├─ 192.168.1.105 Samsung Electronics
├─ scanning...

The task tree lives above the growing network map. Completed tasks show a one-line summary of what was found. The network tree grows in place below — new hosts appear as they're discovered, then ports and vendor info fill in when deeper scans complete. When everything finishes, the full report renders below (or replaces the progress view if --clean is passed).

Event Design (for future C)

Even in Approach A, the scanners will emit structured events internally rather than just returning final results. This doesn't change the scanner interfaces dramatically — each scanner still returns its typed result, but the collector wraps them to emit progress events that listr2 and the network tree consume. When we move to C, these same events get exposed externally as NDJSON.

{ "type": "scanner:start", "scanner": "host-discovery" }
{ "type": "host:found", "ip": "192.168.1.100", "mac": "aa:bb:cc:dd:ee:ff" }
{ "type": "host:enriched", "ip": "192.168.1.100", "vendor": "Apple Inc" }
{ "type": "scanner:complete", "scanner": "host-discovery", "summary": "7 hosts" }
{ "type": "port:found", "ip": "192.168.1.100", "port": 22, "service": "ssh" }
{ "type": "scan:score", "score": 8.2 }
