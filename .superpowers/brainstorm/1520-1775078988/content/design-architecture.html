Component Architecture

How the pieces connect. The key change is the EventEmitter in the collector — everything else is a consumer.

Data Flow
                    ┌──────────────────────┐
                    │     CLI / Dashboard   │
                    │   (triggers a scan)   │
                    └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │      Collector        │
                    │  (orchestrates scan)  │
                    │                       │
                    │  EventEmitter mixin   │◄── NEW: emits scan events
                    └──────────┬───────────┘
                               │ events
              ┌────────────────┼────────────────┐
              │                │                 │
   ┌──────────▼─────┐  ┌──────▼───────┐  ┌─────▼──────────┐
   │  Terminal UI    │  │  Dashboard   │  │  NDJSON stdout  │
   │                 │  │  (via SSE)   │  │  (future C)     │
   │  listr2 tasks   │  │              │  │                 │
   │  network tree   │  │  topology    │  │  pipe to jq,    │
   │  tables         │  │  progress    │  │  SIEM, etc.     │
   │  sparklines     │  │  gauges      │  │                 │
   └─────────────────┘  └──────────────┘  └─────────────────┘
              │                │
              │         ┌──────▼───────┐
              └────────►│    Store     │
                        │ ~/.wifi…/    │
                        │ scans/*.json │
                        └──────────────┘

The collector gains an EventEmitter mixin. Each scanner still returns its typed result as before, but the collector wraps each call to emit scanner:start, scanner:complete, and domain-specific events like host:found. The three consumers (terminal, dashboard, future NDJSON) subscribe independently. This is the "design for C" part — in Approach A we only build the terminal and dashboard consumers, but the event interface is ready for Phase 6.

New Dependencies

listr2 Scan progress with spinners and nested tasks ~28M weekly cli-table3 Responsive Unicode tables with colour support ~19M weekly figures Unicode symbols with ASCII fallbacks ~30M weekly log-update Overwrite previous output for live network tree ~17M weekly sparkly Inline sparkline charts ~200K weekly terminal-link Clickable hyperlinks in supported terminals ~14M weekly

All are pure ESM, well-maintained, and have no native dependencies. Total addition is lightweight.

What Does NOT Change

Scanner implementations stay the same — they still return typed results. The Zod schemas are unchanged. The JSON reporter is unchanged. The analyser and persona modules are unchanged. The store format is unchanged. The existing --skip-* flags and output options all work as before. OTEL instrumentation is unchanged. The TV controller is unaffected.
