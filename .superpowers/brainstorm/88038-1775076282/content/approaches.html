Terminal Revamp — Three Approaches

Each builds on the previous. Pick the approach that matches the ambition level.

A

Progressive Enhancement

Keep the print-and-scroll model, but make it live and accessible.

  • listr2 task list shows scan progress with spinners
  • cli-table3 replaces hand-rolled padding for host/port/compliance tables
  • Network tree grows live as hosts are discovered
  • Dynamic terminal width (goodbye fixed 72 chars)
  • Accessible-by-default colour palette (no red/green only distinctions)
  • Dashboard gets a "Run Scan" button → spawns CLI, streams output via SSE
Effort: Small — incremental changes to existing reporter
Risk: Low — no architectural change, each piece ships independently
B

Ink TUI Mode

React-based terminal UI alongside the existing static report.

  • Everything in A, plus:
  • Ink (React for CLI) renders a multi-pane live view
  • Real-time network tree with device counts, status indicators
  • Inline sparklines for signal, channel saturation, traffic
  • Scanner progress, host table, and scorecard update simultaneously
  • --tui flag for interactive mode, default remains static report
Effort: Medium — new rendering layer, existing scanners unchanged
Risk: Medium — Ink dependency, React paradigm shift for CLI
C

Unified Streaming Architecture

Event-driven scan pipeline feeding both terminal and dashboard in real time.

  • Everything in A + B, plus:
  • Scanners emit events (host-found, port-scanned, analysis-complete)
  • Terminal and dashboard are both consumers of the same event stream
  • Dashboard gets a live topology graph (D3/force-directed) updating as events arrive
  • NDJSON event output for piping into external tools (aligns with Phase 6)
  • Foundation for the watch mode in Phase 6
Effort: Large — refactors the scan pipeline into event emitters
Risk: Higher — touches core architecture, but sets up Phase 6 cleanly

Recommendation

Start with A, design for C. Build the progressive enhancements now — they ship fast and improve the experience immediately. But structure the work so the scanner output can become event-based later. This means the listr2 integration and the live network tree naturally evolve into event consumers when you're ready for Phase 6. Approach B (Ink TUI) becomes an optional add-on at any point.
